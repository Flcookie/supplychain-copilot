# SupplyChain Copilot 改进方案

> 基于 2026-05-08 面试反馈整理。核心方向：从“能回答”升级为“可验证、可追溯、可评估、可持续优化”的企业级供应链 Copilot。

## 1. 总体改进目标

当前系统已经具备 Policy RAG、KPI SQL 查询、Scenario 分析和 LangGraph 路由能力，但面试反馈指出，企业级 Copilot 不能只追求自然语言回答效果，还必须强调以下能力：

- 数据查询和分析结果必须强调准确性与溯源，不能只给结论。
- 同行对比、指标分析不能依赖模型自由生成，必须基于真实数据库、指标口径和可复查计算过程。
- RAG 评估需要系统化，覆盖自动化评测、人工标注和分层指标。
- 检索层需要从单一路径向 hybrid search、多路召回、rerank 演进。
- chunk 策略需要和业务场景绑定，而不是所有文档采用统一切分套路。

改进后的目标是：每个回答都能说明“结论是什么、依据来自哪里、计算过程是什么、可信度如何、哪些地方需要人工复核”。

## 2. 准确性与溯源：从结论型回答升级为证据型回答

### 2.1 现状问题

当前系统已经在 Policy QA 中展示 retrieved documents，在 KPI 路径中记录 SQL 和结果，但证据结构还不够统一：

- Policy 回答有来源，但来源粒度可能停留在文件名或 chunk 层。
- KPI 回答会展示 SQL，但对指标口径、时间范围、数据版本、样本量说明不足。
- Scenario 分析容易混合真实数据和模型推理，用户可能无法区分“已查询事实”和“推理建议”。

### 2.2 改进方案

建立统一的 `Evidence Contract`，让所有节点输出结构化证据，而不是只输出自然语言答案。

建议字段：

```json
{
  "answer": "面向用户的结论",
  "evidence_type": "document | sql | hybrid | simulation",
  "sources": [
    {
      "source_id": "policy_demo.txt#chunk_003",
      "source_name": "supplier_policy_demo.txt",
      "section": "Strategic Supplier Criteria",
      "retrieval_score": 0.82
    }
  ],
  "sql": {
    "query": "SELECT ...",
    "row_count": 3,
    "metric_definition": "OTD = on_time_orders / total_orders",
    "time_range": "last_3_months",
    "data_snapshot": "2026-05-08 demo SQLite"
  },
  "assumptions": [
    "未指定时间范围时默认使用最近 3 个月"
  ],
  "limitations": [
    "当前为 demo 数据，不能代表真实供应商表现"
  ]
}
```

前端回答区也应按固定结构展示：

- 结论：直接回答用户问题。
- 证据：展示引用文档、SQL、返回行数、指标口径。
- 计算说明：解释指标如何得出。
- 限制：说明样本量、时间范围、demo 数据边界。
- 可追溯入口：支持展开查看原始 chunk 或 SQL 结果表。

### 2.3 验收标准

- 100% 的 KPI 回答必须展示 SQL、指标口径、行数和时间范围。
- 100% 的 Policy 回答必须展示文档名、章节或 chunk id。
- Scenario 回答必须区分“数据库查询事实”和“模型建议”。
- 当证据不足时，系统必须明确回答“不足以得出结论”，而不是补全式生成。

## 3. 同行对比与指标分析：从模型生成升级为真实数据驱动

### 3.1 现状问题

供应商对比、趋势分析、风险排序这类问题，如果直接交给 LLM 生成，很容易出现：

- 编造供应商排名。
- 编造不存在的指标。
- 忽略指标定义差异。
- 对样本量很小的数据做过度解读。

这类能力应该由确定性数据查询和指标计算负责，LLM 只负责解释，不负责创造数字。

### 3.2 改进方案

将 KPI 能力拆成三层：

1. 指标解析层  
   从用户问题中解析实体、指标、时间范围、聚合维度和比较对象。

2. 指标计算层  
   使用 SQL 或指标服务计算真实结果。所有指标来自 `metrics_dictionary.json` 或企业指标平台，不允许模型临时定义。

3. 解释生成层  
   LLM 只基于查询结果生成业务解释，并必须引用计算结果。

建议扩展 `metrics_dictionary.json`：

```json
{
  "on_time_rate": {
    "business_name": "准时交付率",
    "formula": "on_time_orders / total_orders",
    "required_tables": ["suppliers", "purchase_orders"],
    "dimensions": ["supplier", "country", "material", "month"],
    "minimum_sample_size": 10,
    "default_time_range": "last_3_months"
  }
}
```

对同行对比类问题，系统应强制走数据查询：

- “Compare Alpha and Beta” -> side-by-side SQL。
- “Top suppliers by OTD” -> ranking SQL。
- “Which supplier is risky” -> 先计算延迟率、订单量、战略供应商标记，再由 LLM 解释。

### 3.3 验收标准

- 对比、排名、趋势类问题禁止纯 LLM 回答。
- 每个指标必须映射到指标字典中的标准口径。
- 当用户请求不存在的指标时，系统应提示可用指标，而不是临时生成。
- 当样本量低于阈值时，回答必须标记为“样本不足，仅供参考”。

## 4. RAG 评估体系：从零散测试升级为系统评测

### 4.1 现状问题

当前项目已经有 router eval，但 RAG 本身还需要更系统的评估。RAG 评估不能只看最终答案是否“看起来合理”，需要拆成检索、引用、答案、稳定性多个层面。

### 4.2 自动化评测

建立 `eval/datasets/rag_eval.json`，每条样本包含：

- user_question：用户问题。
- expected_answer_points：答案应覆盖的关键点。
- expected_sources：应召回的文档或章节。
- answerability：是否可回答。
- intent：policy_qa / kpi_query / scenario_analysis / hybrid。
- difficulty：easy / medium / hard。

自动化指标建议：

- Retrieval Recall@K：正确来源是否出现在 Top K。
- MRR：正确来源排名是否靠前。
- Citation Precision：引用的来源是否真的支持回答。
- Faithfulness：回答是否被证据支持。
- Answer Completeness：关键点覆盖率。
- Refusal Accuracy：证据不足时是否正确拒答。
- Latency：检索和生成耗时。

### 4.3 人工标注

建立人工评审表，对真实或模拟业务问题进行打分：

| 维度 | 评分 | 说明 |
| --- | --- | --- |
| 事实准确性 | 1-5 | 是否与证据一致 |
| 溯源充分性 | 1-5 | 是否给出可复查来源 |
| 指标口径清晰度 | 1-5 | 是否说明公式、时间范围、样本量 |
| 业务可用性 | 1-5 | 回答是否能支持决策 |
| 风险提示 | 1-5 | 是否说明限制和不确定性 |

人工标注结果应沉淀为 regression dataset，用于后续版本回归测试。

### 4.4 分层指标

RAG 评估应分层，而不是只看最终答案：

- 路由层：intent accuracy、ambiguity detection accuracy。
- 检索层：Recall@K、MRR、重复 chunk 比例。
- 重排层：rerank 后正确证据排名提升。
- 生成层：faithfulness、引用准确率、拒答准确率。
- 产品层：任务完成率、用户追问率、人工复核通过率。

### 4.5 验收标准

- 新增 RAG eval 数据集，至少覆盖 30 个 policy 问题、20 个 KPI 问题、10 个 hybrid 问题。
- 每次改动 prompt、retriever、chunk 策略后自动跑评测。
- 关键指标低于阈值时阻止合并，例如 Retrieval Recall@5 < 85%。
- 每轮人工评审保留标注记录和失败案例分析。

## 5. 检索层增强：Hybrid Search、多路召回与 Rerank

### 5.1 现状问题

当前 `rag/retriever.py` 主要使用向量检索，`k=5` 固定。单一路径检索对以下问题不够稳：

- 用户使用精确术语、政策编号、供应商名称时，关键词检索可能更可靠。
- 用户问题较长或包含多个子问题时，单次 embedding 容易稀释重点。
- Top K 固定会导致简单问题召回过多、复杂问题召回不足。

### 5.2 Hybrid Search

建议引入向量检索 + BM25/关键词检索：

- Vector Search：适合语义相近、表达不完全一致的问题。
- Keyword Search：适合政策编号、供应商名称、合同条款、专有名词。
- Metadata Filter：按部门、文档类型、国家、供应商、有效日期过滤。

合并策略：

```text
final_score = 0.6 * vector_score + 0.3 * keyword_score + 0.1 * metadata_boost
```

### 5.3 多路召回

针对不同问题类型做 query expansion 和多路检索：

- 原始问题召回：保留用户原始表达。
- 关键词改写召回：抽取 supplier、metric、policy term。
- HyDE 召回：先生成一个假设答案，再用假设答案检索。
- Section-level 召回：优先召回标题、章节摘要和 policy clause。

多路召回后进行去重、归一化打分和 rerank。

### 5.4 Rerank

引入 reranker 对候选 chunk 进行二次排序。重排输入应包含：

- 用户问题。
- chunk 文本。
- chunk 标题、来源、更新时间。
- 与问题相关的实体匹配情况。

Rerank 后只把最相关、非重复、证据充分的 chunk 传给 LLM，降低上下文噪声。

### 5.5 验收标准

- 检索日志记录每一路召回结果、原始分数、融合分数和 rerank 分数。
- 对政策编号、供应商名称、精确条款类问题，Keyword Recall 明显提升。
- 对语义改写问题，Vector Recall 保持稳定。
- Rerank 后正确来源进入 Top 3 的比例提升。

## 6. Chunk 策略：从统一切分升级为场景化切分

### 6.1 现状问题

统一 chunk size 和 overlap 适合 PoC，但不适合企业知识库。不同文档结构和业务场景需要不同 chunk 策略：

- 政策制度需要保留章节层级和条款完整性。
- 合同条款需要按 clause 切分，避免跨条款混淆。
- KPI 指标说明需要公式、口径、字段映射完整保留。
- SOP 流程需要按步骤和角色切分。
- FAQ 可以按 question-answer pair 切分。

### 6.2 场景化策略

建议按文档类型配置 chunk 策略：

| 文档类型 | 切分单位 | Metadata | 适用问题 |
| --- | --- | --- | --- |
| Policy | 章节 + 条款 | policy_name, section, effective_date | 规则解释、合规问答 |
| Contract | clause | supplier, contract_id, clause_type | 合同义务、违约责任 |
| KPI Dictionary | 指标项 | metric_id, formula, owner | 指标口径、字段解释 |
| SOP | 步骤 + 角色 | process, role, step_no | 操作流程、责任边界 |
| FAQ | Q&A pair | topic, owner | 高频问题 |

### 6.3 Chunk Metadata

每个 chunk 应包含足够 metadata：

```json
{
  "doc_type": "policy",
  "source_name": "supplier_policy_demo.txt",
  "section_title": "Strategic Supplier Criteria",
  "chunk_id": "policy_demo_003",
  "effective_date": "2026-01-01",
  "owner": "Procurement",
  "access_level": "internal",
  "language": "en"
}
```

这些 metadata 应用于：

- 检索过滤。
- 来源展示。
- 权限控制。
- 过期文档降权。
- 评估数据对齐。

### 6.4 验收标准

- 每类文档有独立 chunk 策略配置。
- chunk 不切断公式、表格、条款和 Q&A pair。
- 每个回答可以展示 chunk 的章节来源。
- 通过 RAG eval 比较不同 chunk 策略的 Recall@K 和 citation precision。

## 7. 分阶段落地路线

### Phase 1：补齐证据结构和展示

- 为 Policy、KPI、Scenario 三条路径统一 `Evidence Contract`。
- KPI 回答强制展示 SQL、指标口径、样本量和时间范围。
- Scenario 回答区分事实、假设和建议。
- UI 增加“证据详情”展开区。

### Phase 2：建立评测闭环

- 新增 RAG eval dataset。
- 实现自动化评测脚本。
- 建立人工标注表和失败案例库。
- 将 router eval 和 RAG eval 合并为版本化评测报告。

### Phase 3：增强检索

- 引入 BM25 或关键词索引。
- 实现 vector + keyword + metadata 的 hybrid fusion。
- 增加多路召回和 rerank。
- 记录检索链路日志，支持错误分析。

### Phase 4：重构 ingestion 和 chunk 策略

- 按文档类型配置 chunker。
- 增加结构化 metadata。
- 对 policy、contract、KPI dictionary、SOP 分别建索引或 namespace。
- 用评测结果选择最优 chunk 参数。

## 8. 面试答辩表达建议

可以将优化思路概括为：

> “我会把这个 Copilot 从 PoC 级的自然语言回答系统，升级为企业可用的 evidence-grounded decision assistant。核心不是让模型直接给结论，而是让模型编排数据查询、检索证据和业务解释。对于 KPI 和同行对比，我会强制走真实数据和指标字典；对于 RAG，我会用自动化评测、人工标注和分层指标持续优化；对于检索，我会引入 hybrid search、多路召回和 rerank；对于 chunk，我会按 policy、contract、KPI dictionary、SOP 等场景分别设计，而不是统一切分。”

这套回答能体现三个重点：

- 产品意识：企业用户需要可信、可复核的答案。
- 工程意识：通过 SQL、指标字典、Evidence Contract、评测脚本降低幻觉。
- RAG 专业度：把检索、重排、chunk、评估拆开优化，而不是只调 prompt。
